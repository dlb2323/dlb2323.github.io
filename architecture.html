Introduction to our Diagrams

To describe how we plan and implement different functions of the game, we have chosen to use UML diagrams. Structural Diagrams will visualize core entities, their features, and relationships - covering the static portion of the project. In contrast Behavioural Diagrams include sequence diagrams covering how components interact with each other. We will be creating Structural Class diagrams and State Machine diagrams, two diagrams of the main UML diagram categories. We chose to use these specific diagrams as they are the most commonly used diagrams, meaning they will be widely understood and have plenty of resources available.

For an Architecture Pattern, we will be using Entity-Component-Systems Diagram. This is a favoured method in video game development, we chose it as it facilitates code reusability allowing us to save time in the production phase of this project. This is important with the short amount of time we have. It is an appropriate use case as a lot of the built-in LibGDX classes have to be implemented using composition, such as TextureRegion, Animation and OrthographicCamera. We will be using Inheritance for some of the Entities, for example LivingThing inherits from Entity. This stops code repetition as the parent class will have a method defined, one example in our code is the fact Entity has the draw method (placing the Entities onto the screen) defined, meaning it doesn’t have to be repeated in the Enemy and Player class.

Our thoughts when designing the Diagrams

We began by considering the main objects of the game: the Player and the Enemies. These are the principle moving parts of the game. They will share some common methods (such as collision, kill and move) and attributes (such as texture, position, speed). Therefore, we have decided to put them all in the same parent class, reducing code repetition. There will be more in common between the Player and Enemies than between them and a Projectile, for example a Projectile does not have HP. So we have designed an abstract sub-class that contains these extra functionalities, which will be the parent class of the Player and Enemy classes, named LivingThings.

Over the course of this project, we had three interim design phases, creating diagrams for our thoughts in the beginning, middle and end.

On our first approach we just prioritised the Entity Class, and the two derived classes Player, and Enemy. On the final diagram, we chose to include Projectiles as a class, inheriting from Entity - this meant we had to include a new derived class to Entity, LivingThings. This will be the new parent class to Player and Enemy, a decision made because they share more attributes (for example dmg and hp) and methods (such as Move and Attack). Our thought process for this initial decision was that the 3 classes (Enemy, Player and Projectile) will share a number of attributes and functions. For example, the collision method, which is implemented in the superclass Entity, will be used by all three classes. All entities will have the same collision detection method, meaning there’s no need to have duplicated methods.

The final UML Diagram we made during this phase:

Abstract Class Entity

Method Purpose
Collision An implemented method, assuming that all objects are the same shape, using the same collision detection programme to determine whether a collision has occurred with the local object and another entity or tile. Returns the object/tile that the collision occurred with.
Kill An implemented method that removes the object from the entities array and destroys its instance. Returns true.
Move An unimplemented method that will be defined differently in each subclass - for example, a projectile will just move straight from where an enemy is facing until it hits something, whereas an enemy will move toward the player and keep facing it at all times.
Position The coordinates of the entity on the map.
Sprite Contains the sprite information of the entity.
Speed Contains the velocity that the entity moves.

Class Projectile

Method Purpose
Move using information parsed from the object that defined it (for example, an instance of enemy) like the direction it is facing, the projectile will move in a constant direction at a constant speed until it hits something. It will then call the Collide method and, if it hit a player, for example, it would run the Impact method.
Impact called only when a projectile collides with another entity. If it hit the player, for example, the projectile would run Kill on itself, then it would deal the defined amount of damage ‘dmg’ to the player. If it hit another projectile, both would be killed.
