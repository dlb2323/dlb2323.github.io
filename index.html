Single Statement of Need

Our Client has requested a single-player game which allows the player to escape a maze, wrapped in the guise of a University. There are three primary objectives for this project which have been outlined by our customer: The user should be able to escape within five minutes; There should be multiple events to interact with, some of which are hidden; There should be an opposing force working against the user.

However, as developers, we recognise that there are many additional requirements for this project. As such, we arranged a meeting with our client to discuss and gather these additional requirements. A prior brainstorming session led us to compiling a list of questions to ask our client, allowing us to delve into the functionalities and wishes of the customer. This meeting was valuable to discovering the needs of our client, giving a clearer understanding of customer-needs and specifications. In the long-term, continuing to discuss and collaborate with our Client will aid in avoiding customer frustration and misunderstandings; also aligning with an agile approach to software development, prioritising customer collaboration.

While this customer research gave us an insight into User Requirements, we understand that there is a hierarchy of requirement types and complexity. To level this complexity, we have separated our requirements into User Requirements and System Requirements. The latter is further divided into Functional, Non-Functional, and Constraint Requirements. This gives a simple classification of key requirements, which in themselves are given an appropriate Requirement ID. This allows us to easily reference and link requirements when needed. We intend for our system to meet all requirements that ‘shall’ be done in our specification. This ensures our final project meets as many of our clients primary objectives as possible.

In summary: using the requirements elicited from the client, we will create a Maze Game based in a University, where the user should be able to escape within five minutes. After consulting our clients, as developers we have chosen to include multiple variants of boons and obstacles to interact with, as well as incorporating an environmental narrative to capture the interest of users.

User Requirements

User Requirements outline the general features and tasks the users should be able to carry out within the game.

UR_COMPLETE_ON_TIME The game should be balanced in difficulty for prospective students of the university (16+) to complete in 5 minutes, on their first try. Certain events could hinder or benefit the player’s completion. Shall
UR_CAMPUS The map should include inside and outside areas, including; Lecture theatres, a lake, the lake underpass, bridges, different buildings like central hall and derwent. Lots of time should be spent on the map, and should be done ourselves. Should

System Requirements

System requirements describe how the game will meet the needs of prospective players from a technical point of view. They will describe how we deliver key functionalities, such as player movement across the map, event completion, and interaction with items that affect the player’s progress.

Each System Requirement supports a user requirement — for example, balancing gameplay difficulty and designing the map both contribute to allowing players to complete the game on time. This allows us to keep track of User Expectations, and supports game development, ensuring that it runs smoothly.

Functional Requirements

ID Description User Requirements
FR_REQUIRENAME (Description & Fit Criterion) UR_REFERENCE
FR_COMPLETION_TIME The system shall include a time that limits the gameplay up to five minutes only. When time runs out, the game will automatically end. UR_COMPLETE_ON_TIME
FR_DIFFICULTY_BALANCE The system shall try to balance the game difficulty such that even an amateur player is able to finish the game in the given time period. UR_COMPLETE_ON_TIME
FR_MAP_STRUCTURE The system will include an explorable map of the entire campus giving access to both indoor and outdoor areas of the buildings and the campus. UR_CAMPUS
FR_PLAYER_NAVIGATION The system shall allow the players to move across the campus using keyboard or controller inputs. UR_CAMPUS
FR_INTERACTIVE_ELEMENTS The system shall include interactive elements in the gameplay that might affect the player’s progress which could be either positive or negative. UR_BOONS_AND_OBSTACLES
FR_GAME_DIMENSION The system shall present the game in a 2D environment using tile maps. UR_DIMENSION
FR_MULTIPLE_ENDINGS The system shall provide the user with multiple winning conditions that all require gaining specific objectives UR_WIN
FR_PROGRESS_TRACKING The system shall track the player’s progress and when the winning conditions are met, it would trigger the end of the game. UR_WIN
FR_PROCEDURAL_GENERATION The system shall procedurally generate the placement of different rooms and items to make each gameplay session unique while maintaining the main structure of the campus unchanged. UR_MAP_GENERATION
FR_COMBAT_MECHANISM The system shall include a combat mechanism, when if implemented, it would use only non violent elements to maintain the PG rating. UR_COMBAT
